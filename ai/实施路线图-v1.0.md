# WP Magick Toolbox 实施路线图

> **版本**: 2.0.84 → 目标 2.1.0+  
> **定位**: 面向中国 WordPress 站长的一站式实用工具箱插件  
> **核心原则**: 先治技术债务，再提用户体验，最后添生态功能  
> **总周期预估**: 10-12 周（约 3 个月）  

---

## 实施原则

1. **不做新功能，直到技术债务清理完成** - 继续堆功能会让插件不可维护
2. **每个阶段必须有明确的验收标准** - 不验收不进入下一阶段
3. **保持向后兼容** - 老用户升级不能丢失配置
4. **优先解决用户痛点，而非开发者兴趣** - 外观特效暂停新增

---

# 第一阶段：技术债务清理（第 1-3 周）

**目标**: 解决阻碍插件长期发展的核心技术问题，建立可维护的架构基础。

**核心原则**: 本阶段**不开发任何新功能**，只做架构优化和现有问题修复。

## 1.1 配置存储拆分 [P0]

### 现状问题
- 所有 90+ 功能的配置存储在单个 `wp_options` 键 `Magick_ToolBox_Option` 中
- JSON 体积过大，保存/读取性能下降
- 并发保存可能导致数据丢失
- 无法按需加载特定模块配置

### 目标方案
按模块拆分为多个 Option 键：

```
Magick_ToolBox_Option_Core      // 核心设置
Magick_ToolBox_Option_SEO       // SEO 功能
Magick_ToolBox_Option_Page      // 页面功能
Magick_ToolBox_Option_Media     // 媒体优化
Magick_ToolBox_Option_Comment   // 评论功能
Magick_ToolBox_Option_Security  // 登录安全
Magick_ToolBox_Option_Appearance // 外观特效
Magick_ToolBox_Option_Stats     // 统计分析
```

### 实施步骤
1. **设计新配置结构** - 定义每个模块包含的字段
2. **编写迁移脚本** - 自动将旧配置拆分到新键
3. **保留备份机制** - 迁移前自动备份旧配置到 `Magick_ToolBox_Option_Backup`
4. **更新读取逻辑** - `MaBox_Admin::get_config()` 支持多键读取
5. **更新保存逻辑** - React 端按模块保存
6. **版本标记** - 记录配置版本，方便后续升级

### 验收标准
- [ ] 旧配置成功迁移，无数据丢失
- [ ] 迁移失败可自动回滚
- [ ] 所有现有功能正常工作
- [ ] 保存速度提升（主观感受）

## 1.2 模块注册表机制 [P0]

### 现状问题
- 每个功能需要手动在 `index.php` 中 `require_once` 和 `::run()`
- 新增功能需要改多个文件
- 无法做依赖检测、冲突检测
- 无法统计已启用功能数量

### 目标方案
建立统一的模块注册表：

```php
// admin/modules/registry.php
return [
    'seo.home_tdk' => [
        'class'     => 'Npcink_SEO_Home_TDK',
        'file'      => 'admin/partials/seo/home-tdk.php',
        'option_key'=> 'seo.home_tdk.enabled',
        'category'  => 'seo',
        'scope'     => 'frontend', // frontend / admin / both
        'dependencies' => [],      // 依赖的其他模块
        'theme_requirement' => '', // 如 'b2'
        'risk_tags' => ['recommended', 'seo'], // 风险标签
    ],
    // ... 其他模块
];
```

### 实施步骤
1. **设计注册表结构** - 定义完整的模块元数据格式
2. **创建注册表文件** - 将所有现有功能录入注册表
3. **编写模块加载器** - 统一读取注册表、检查开关、按需加载
4. **重构现有加载逻辑** - 替换所有手动 `require_once`
5. **支持依赖检测** - 如 B2 商城统计依赖 B2 主题

### 验收标准
- [ ] 所有 90+ 功能通过注册表加载
- [ ] 新增功能只需在注册表添加一条记录
- [ ] 未启用的功能不注册 Hook，不占用内存
- [ ] B2 主题依赖检测正常工作

## 1.3 前端资源按需加载 [P0]

### 现状问题
- React 构建的后台 JS/CSS 可能在所有后台页面加载
- 前台特效资源未严格按需控制
- 移动端不适配，但资源仍然加载

### 目标方案
```
后台资源:
  - 仅在插件设置页面加载 React 应用
  - 其他后台页面完全不加载

前台资源:
  - 功能未开启 → 不加载资源
  - 当前页面不需要 → 不加载资源
  - 移动端不需要 → 不加载资源

具体示例:
  - 仅开启点击特效时才加载点击特效 JS
  - 仅开启背景特效时才加载背景特效资源
  - 仅文章页才加载文章评分 JS
  - 仅评论页才加载评论增强 JS
  - 仅登录页才加载登录页美化 CSS
```

### 实施步骤
1. **后台资源隔离** - 检查 `is_page('Magick_ToolBox_Option')` 再加载
2. **前台资源梳理** - 列出所有功能及其依赖的资源文件
3. **添加加载条件** - 每个功能的 `run()` 方法中检查开关后再 `wp_enqueue`
4. **页面级控制** - 添加 `is_home()`, `is_single()`, `is_page()` 等条件
5. **设备级控制** - 使用 `wp_is_mobile()` 做移动端判断

### 验收标准
- [ ] 在未打开插件设置页时，后台不加载插件 JS/CSS
- [ ] 关闭所有功能时，前台不加载任何插件资源
- [ ] 仅开启首页 TDK 时，前台不加载特效资源
- [ ] 使用浏览器 DevTools 的 Network 面板验证

### 实施步骤

3. **实现过滤逻辑** - 根据输入实时过滤显示的功能卡片
4. **添加跳转定位** - 点击结果自动切换到对应 Tab 并滚动到位置

### 验收标准
- [ ] 搜索"SEO"能显示所有 SEO 相关功能

## 1.5 功能风险标签 [P1]

### 现状问题
- 用户无法判断功能的影响范围
- 新手可能误开启影响性能或 SEO 的功能
- 缺乏专业感

### 目标方案
为每个功能添加标签：
```
[推荐]      - 安全且建议开启
[SEO]       - 影响搜索引擎优化
[安全]      - 提升网站安全性
[性能]      - 可能影响网站性能
[谨慎]      - 需要谨慎开启
[仅前台]    - 仅在前台生效
[仅后台]    - 仅在后台生效
[需主题兼容] - 需要特定主题支持
```

### 实施步骤
1. **定义标签规范** - 确定标签类型和样式
2. **为所有功能打标签** - 在模块注册表中添加标签字段
3. **React 端显示标签** - 在功能卡片上展示彩色标签
4. **添加标签过滤** - 支持按标签筛选功能

### 验收标准
- [ ] 每个功能至少有一个标签
- [ ] 标签颜色区分明显
- [ ] 支持按标签筛选
- [ ] 鼠标悬停显示标签说明

## 3.1 备案与合规信息 [P0]

### 价值
- 国内所有正规网站必须展示备案号
- 工信部会抽查，不合规可能关站
- 很多主题不自带合规页脚
- 实现简单，价值高

### 功能清单
1. **底部备案号自动添加**
   - ICP 备案号（如：京ICP备12345678号）
   - 公安网备号（如：京公网安备11010102001234号）
   - 自动插入到页脚（wp_footer 钩子）
   - 支持自定义位置

2. **备案号悬浮核验链接**
   - 点击备案号跳转到工信部查询页面
   - 自动拼接查询链接

3. **隐私政策/Cookie 同意弹窗**
   - 多款弹窗样式模板
   - 支持自定义文案
   - 记录用户同意状态
   - 支持「仅显示一次」

4. **网站底部版权信息模板**
   - 预设版权模板
   - 支持自定义 HTML
   - 自动更新年份

### 实施步骤
1. **设计设置面板** - 在「页面外观」或新增「合规信息」分组
2. **实现备案号插入** - 使用 `wp_footer` 钩子
3. **实现 Cookie 弹窗** - 前端 JS + 后端记录状态
4. **提供样式模板** - 3-5 款预设样式

### 验收标准
- [ ] 备案号正确显示在页脚
- [ ] 点击备案号可跳转到查询页面
- [ ] Cookie 弹窗可正常显示和关闭
- [ ] 支持自定义样式和文案

## 3.2 百度收录推送套件 [P0]

### 价值
- 百度是国内站长最看重的流量来源
- 当前插件仅有统计代码，缺少推送能力
- 与"多合一搜索推送"等 20 万+ 下载插件竞争的核心能力
- **注意**：百度搜索资源平台 API 近期有变更，需确认最新接口

### 功能清单
1. **文章发布主动推送**
   - 通过 `publish_post` 钩子触发
   - 文章发布时自动推送到百度
   - 显示推送结果（成功/失败）

2. **自动推送 JS 代码注入**
   - 在页面底部插入百度自动推送 JS
   - 访客访问时自动推送页面

3. **批量推送历史文章**
   - 选择时间段，批量推送历史文章
   - 使用 WP Cron 定时任务分批处理（每次 100 条）
   - 显示推送进度和结果

4. **收录状态查询**
   - 查询文章是否被百度收录
   - 在文章列表显示收录状态图标
   - 支持批量查询

### 实施步骤
1. **确认最新 API** - 查阅百度搜索资源平台最新 API 文档
2. **设计设置面板** - API 接口配置（site + token）
3. **实现主动推送** - `publish_post` 钩子 + curl 请求
4. **实现自动推送 JS** - `wp_footer` 插入 JS 代码
5. **实现批量推送** - 使用 WP Cron + 分批处理
6. **实现收录查询** - 使用 site: 语法或 API 查询

### 验收标准
- [ ] 文章发布后自动推送到百度
- [ ] 推送结果显示在文章编辑器
- [ ] 支持批量推送历史文章
- [ ] 文章列表显示百度收录状态
- [ ] API 错误时有明确提示

## 3.5 登录安全中心 [P1]

### 价值
- WordPress 后台爆破很常见
- 用户愿意为"安全防护"付费
- 当前仅有验证码，缺少防护体系

### 功能清单
1. **限制登录失败次数**
   - 设置最大失败次数（如 5 次）
   - 超过后临时锁定账号

2. **登录失败锁定 IP**
   - 记录失败登录的 IP
   - 超过阈值后封禁 IP（如 30 分钟）
   - 使用 Transient 实现

3. **修改登录入口地址**
   - 自定义 wp-login.php 地址
   - 如：将 `wp-login.php` 改为 `my-secret-login`
   - 原地址自动跳转到 404 或首页

4. **禁止用户名枚举**
   - 阻止通过 REST API 或 ?author=1 枚举用户名

5. **登录通知邮件**
   - 新设备/新 IP 登录时发送邮件通知

6. **后台登录日志**
   - 记录所有登录尝试（成功/失败）
   - 显示 IP、时间、UA
   - 支持导出日志

7. **仅允许指定 IP 访问后台**
   - 白名单机制
   - 非白名单 IP 访问后台时自动跳转

### 实施步骤
1. **整合现有登录功能** - 将验证码、防水墙等整合到安全中心
2. **实现登录限制** - `wp_login_failed` 钩子 + Transient
3. **实现登录地址修改** - `login_url` 过滤器
4. **实现日志系统** - 自定义表存储登录日志

### 验收标准
- [ ] 登录失败 5 次后锁定 IP
- [ ] 自定义登录地址生效
- [ ] 用户名枚举被阻止
- [ ] 登录日志完整记录
- [ ] 白名单机制正常工作

## 4.1 AI 审核引擎 [P1]

### 定位
用 AI 做内容安全审核，而非内容生成。中国站长对内容合规压力大，AI 审核是刚需。

### 架构设计
```
AI Provider 抽象层（同一时间仅激活一个）
├── DeepSeek（默认，性价比高，国内稳定）
├── 阿里云内容安全（可选，企业用户偏好）
├── 自定义 API（用户自行配置 endpoint）
└── 本地规则引擎（无 API 时自动降级）
```

### 功能清单
1. **评论 AI 审核**
   - 自动识别广告、灌水、敏感内容
   - 命中后评论进入待审核或直接屏蔽
   - 支持严格/宽松模式切换

2. **图片 AI 审核**（可选，视 API 成本而定）
   - 上传时自动识别违规图片
   - 支持批量审核历史图片

3. **审核日志**
   - 记录 AI 判断结果及置信度
   - 支持人工复核（误判纠正）
   - 审核统计面板

4. **Provider 管理**
   - 后台配置 API Key 和 Endpoint
   - 同一时间仅启用一个 Provider
   - 切换时自动保存上次配置
   - 无 API 时自动降级到本地规则引擎

### 实施步骤
1. **设计 Provider 抽象层** - 统一接口，支持多 Provider 切换
2. **实现 DeepSeek 对接** - 评论审核 Prompt 工程
3. **实现本地规则引擎** - 关键词+正则作为降级方案
4. **设计审核面板** - 审核日志、统计、复核入口
5. **实现 Provider 配置 UI** - API Key 管理、模式切换

### 验收标准
- [ ] DeepSeek 审核评论生效
- [ ] 支持严格/宽松模式
- [ ] 无 API 时自动降级到本地规则
- [ ] 审核日志可查看和复核
- [ ] 同一时间仅一个 Provider 激活

### 注意事项
- 不碰 AI 内容生成（标题、摘要等），避免 API 成本失控
- AI 仅用于审核基础设施，不做花哨功能
- 后台命名避免技术术语（不用 Prompt、Model 等）
- 使用「AI 审核助手」「智能过滤」等命名

```php
function magick_toolbox_migrate_config() {
    $old_option = get_option('Magick_ToolBox_Option');
    if (!$old_option) return;
    
    // 备份旧配置
    update_option('Magick_ToolBox_Option_Backup_v2084', $old_option);
    
    // 拆分到新键
    $config = json_decode($old_option, true);
    update_option('Magick_ToolBox_Option_SEO', json_encode($config['seo'] ?? []));
    update_option('Magick_ToolBox_Option_Page', json_encode($config['page'] ?? []));
    // ... 其他模块
    
    // 标记已迁移
    update_option('Magick_ToolBox_Option_Version', '2.1.0');
}
```

## 2. 模块注册表加载示例

```php
// admin/class-magick-mixtrue-admin.php
public function load_modules() {
    $registry = require 'admin/modules/registry.php';
    
    foreach ($registry as $module_id => $module) {
        // 检查开关
        $enabled = $this->get_config($module['option_key']);
        if (!$enabled) continue;
        
        // 检查依赖
        foreach ($module['dependencies'] as $dep) {
            if (!$this->is_module_enabled($dep)) continue 2;
        }
        
        // 检查主题要求
        if ($module['theme_requirement'] && !current_theme_supports($module['theme_requirement'])) {
            continue;
        }
        
        // 加载并运行
        require_once $module['file'];
        $module['class']::run($this->get_module_config($module_id));
    }
}
```

## 3. 前端按需加载示例

```php
// admin/partials/page/function/click-effect.php
public static function run($config) {
    self::$option = $config;
    
    // 只在开启时加载资源
    $enabled = MaBox_Admin::get_config(self::$option, 'click_effect.enabled');

public static function enqueue_assets() {
    // 只在文章页加载
    if (!is_single()) return;
    
    wp_enqueue_style('click-effect', MAGICK_TOOLBOX_URL . 'public/css/click-effect.css');
    wp_enqueue_script('click-effect', MAGICK_TOOLBOX_URL . 'public/js/click-effect.js');
}
```

## 5. 免费增值策略

**核心原则**: 全部功能免费开放，通过增值服务盈利。不做功能分层，不做付费墙。

| 策略 | 说明 |
|------|------|
| 功能策略 | 所有功能免费，无 Pro/免费分层 |
| 盈利模式 | 技术支持、定制开发、代部署、主题适配服务 |
| 用户增长 | 降低使用门槛，扩大免费用户基数 |
| 增值服务 | 插件内嵌服务入口，让有需求的用户能找到你 |

### 为什么不做功能分层？

1. **中国 WordPress 站长付费意愿低** - 功能付费墙会直接阻碍用户增长
2. **口碑传播依赖完整体验** - 用户只会推荐"好用且免费"的插件
3. **增值服务利润更高** - 定制开发、代部署的客单价远高于插件订阅
4. **竞争壁垒在生态** - 主题适配、文档、社区才是长期壁垒，不是功能锁
