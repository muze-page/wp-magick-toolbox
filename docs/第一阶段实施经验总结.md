## 二、架构决策与经验

### 2.1 配置存储拆分

#### 决策：按顶层模块拆分为 7 个 Option

**拆分方案**：
```
Magick_ToolBox_Option_Optimize    → optimize.*
Magick_ToolBox_Option_Page        → page.*
Magick_ToolBox_Option_Function    → function.*
Magick_ToolBox_Option_H5          → h5.*
Magick_ToolBox_Option_Login       → login.*
Magick_ToolBox_Option_Shortcode   → shortcode.*
Magick_ToolBox_Option_Template    → template.*
```

**关键经验**：

1. **React 端无需改动**：拆分对前端完全透明，因为 `get_merged_config()` 合并后传给 React 的数据结构与原来完全一致。这是最重要的设计决策——**后端拆分，前端无感**。

2. **迁移脚本必须幂等**：使用 `Magick_ToolBox_Config_Version` 标记迁移状态，避免重复迁移。新安装直接标记为已迁移，跳过迁移逻辑。

3. **备份键名带版本号**：`Magick_ToolBox_Option_Backup_v210` 而非 `Magick_ToolBox_Option_Backup`，方便后续升级时区分不同版本的备份。

4. **导入导出保持格式不变**：`export_config()` 返回合并后的完整配置，导出的 JSON 文件格式与旧版完全一致，老用户导入导出无感知。

#### 踩坑记录

| 问题 | 原因 | 解决 |
|------|------|------|
| REST API 方法重复 | 编辑时只替换了方法体，未删除旧方法 | 使用 `sed` 批量删除残留代码块 |
| `load()` 方法被误删 | 大范围删除时未注意方法边界 | 重新添加并验证 |

### 2.2 模块注册表机制

#### 决策：数据驱动替代硬编码

**旧架构问题**：
- `build_active_modules_map()` 320+ 行 if-else 嵌套
- `activate_module()` 440+ 行 handler map 闭包
- 新增功能需要改 3 个地方：注册表判断 + 路径映射 + handler 闭包

**新架构优势**：
- 新增功能只需在 `registry.php` 添加 1 条记录
- 加载逻辑统一在 `loader.php` 中，零硬编码
- 支持 `always_load`、`mobile_only`、`theme_requirement` 等元数据

#### 注册表字段设计

```php
'module.id' => array(
    'class'             => 'Npcink_Module_Class',  // 必须
    'file'              => 'path/to/file.php',      // 必须，相对于 partials/
    'option_key'        => 'path.to.switch',        // 必须，点分路径
    'category'          => 'optimize',              // 必须，模块分类
    'scope'             => 'frontend',              // frontend/admin/both
    'config_path'       => 'path.to.config',        // 可选，传递给 run() 的配置路径
    'always_load'       => true,                    // 可选，始终加载（短代码/模板）
    'mobile_only'       => true,                    // 可选，仅移动端加载
    'theme_requirement' => 'B2',                    // 可选，需要特定主题
    'risk_tags'         => array('推荐', 'SEO'),    // 可选，风险标签
),
```

#### 关键经验

1. **`option_key` 使用点分路径**：`optimize.site.hide_top_toolbar` 而非 `optimize['site']['hide_top_toolbar']`，方便 `get_nested_value()` 递归解析。

2. **`config_path` 与 `option_key` 分离**：`option_key` 是开关判断（布尔值），`config_path` 是配置传递（完整配置对象）。例如点击特效的开关是 `page.feature.particle`（字符串），但传递给 `run()` 的是 `page.feature`（完整配置对象）。

3. **`always_load` 标记始终加载的模块**：短代码和页面模板需要始终注册，不受开关控制。

4. **依赖检测通过 `theme_requirement` 实现**：B2 主题专属功能通过 `MaBox_Tool::theme_active('B2')` 判断，无需额外依赖系统。

#### 踩坑记录

| 问题 | 原因 | 解决 |
|------|------|------|
| 注册表文件过大（700+ 行） | 90+ 模块全部在一个文件 | 可接受，数据文件非逻辑代码，按分类注释分隔 |
| 部分模块 `run()` 签名不一致 | 有的接受参数，有的不接受 | 通过 `config_path` 字段区分，有则传配置，无则不传 |

### 2.3 功能搜索

#### 决策：前端静态索引 + 实时过滤

**方案对比**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 前端静态索引（最终选择） | 零后端改动，响应快，离线可用 | 需要手动维护索引 |
| 后端 API 搜索 | 自动同步 | 增加请求开销，响应慢 |
| 基于注册表自动生成 | 零维护 | 需要 PHP→JS 数据传递，增加复杂度 |

**选择前端静态索引的理由**：
1. 搜索是纯前端交互，不需要后端参与
2. 90+ 功能索引仅 ~15KB，加载成本极低
3. 支持拼音/关键词扩展，灵活度高

#### 搜索索引设计

```typescript
export interface SearchItem {
  id: string;        // 与注册表 module_id 对应
  label: string;     // 功能中文名
  tabKey: string;    // 目标 Tab 的 key
  tabLabel: string;  // Tab 名称（用于显示标签）
  section?: string;  // 子分类（外观/权限/功能等）
  keywords?: string[]; // 扩展搜索关键词
  tags?: string[];   // 风险标签（推荐/SEO/安全/性能/谨慎）
}
```

#### 关键经验

1. **`id` 必须与 Form.Item 的 `id` 属性对应**：搜索跳转依赖 `document.getElementById(itemId)` 定位，需要在每个 Form.Item 上添加 `id` 属性。

2. **颜色映射集中管理**：`tagColorMap` 统一管理标签颜色，避免硬编码散落在多处。

3. **搜索结果限制 20 条**：避免过长列表影响性能，同时足够覆盖大多数搜索场景。

#### 踩坑记录

| 问题 | 原因 | 解决 |
|------|------|------|
| HeaderBlock 组件未接收 props | 修改了组件签名但未更新调用处 | 添加 `onNavigate` prop 传递 |
| 搜索锚点不生效 | Form.Item 缺少 `id` 属性 | 逐个添加 `id` 属性 |

### 2.4 风险标签

#### 决策：注册表驱动 + React 彩色显示

**标签体系**：

| 标签 | 颜色 | 含义 | 示例功能 |
|------|------|------|----------|
| 推荐 | green | 安全且建议开启 | 移除 WP 版本号、图片 Alt |
| SEO | blue | 影响搜索引擎优化 | 首页 TDK、文章 SEO |
| 安全 | red | 提升网站安全性 | 登录验证码、移除 sitemap 用户 |
| 性能 | orange | 可能影响网站性能 | 背景特效、点击特效 |
| 谨慎 | volcano | 需要谨慎开启 | 禁止复制、禁用自动更新 |
| 仅前台 | purple | 仅在前台生效 | 外观特效类 |
| 仅后台 | cyan | 仅在后台生效 | 后台优化类 |
| 需主题兼容 | gold | 需要特定主题支持 | B2 商城统计 |

#### 关键经验

1. **标签在注册表和搜索索引中双写**：注册表是数据源，搜索索引是展示层。后续可考虑从注册表自动生成搜索索引。

2. **标签颜色语义化**：红色=安全（警示），绿色=推荐（正面），橙色=性能（注意），避免颜色混淆。

### 2.5 Bug 修复经验

#### Bug 1: 文章统计功能不可用

**根因**：CSS/JS 路径计算错误
```php
// 错误：dirname(__DIR__) 只向上一级
$build_css = plugin_dir_url(dirname(__DIR__)) . 'count/dist/index.css';
// 实际路径：admin/partials/ → 需要向上 3 级到插件根目录
```

**修复**：
```php
$build_css = plugin_dir_url(dirname(dirname(dirname(__DIR__)))) . 'vite/count/dist/index.css';
```

**教训**：路径计算应使用 `plugin_dir_path(__FILE__)` 配合相对路径，避免多层 `dirname()` 嵌套。

#### Bug 3: 未登录隐藏分类时下载部分可显示

**根因**：`the_content` 过滤器只修改文章内容，B2 主题的下载区通过独立 hook 渲染，不在 `the_content` 范围内。

**修复**：添加 `wp_footer` hook 注入 CSS 隐藏下载区
```php
add_action('wp_footer', array(__CLASS__, 'hide_download_for_restricted_categories'));
```

**教训**：修改内容展示时，需考虑主题可能通过独立 hook 渲染的区域。

#### Bug 5: WPS 跳转中间页移动端未适配

**根因**：弹窗使用固定定位 `left: calc(50% - 225px)`，小屏幕下超出视口。

**修复**：添加响应式媒体查询
```css
@media (max-width: 768px) {
    .modal-body {
        left: 50% !important;
        top: 50% !important;
        transform: translate(-50%, -50%);
        width: 90% !important;
    }
}
```

**教训**：所有 go 中间页都需要移动端适配，建议统一提取公共响应式样式。

## 三、通用经验总结

### 3.1 大文件重构策略

1. **先读后改**：重构前必须完整阅读文件，理解所有依赖关系
2. **分步删除**：大范围删除代码时，使用 `grep` 确认方法边界，避免误删
3. **即时验证**：每次编辑后立即运行 LSP 诊断，不要积累错误
4. **备份关键逻辑**：删除旧代码前，确保新代码已完全替代其功能

### 3.2 跨层修改注意事项

1. **PHP 后端改动需同步检查**：
   - 所有引用该配置的位置（grep 搜索）
   - 导入导出逻辑
   - REST API 端点
   - AJAX 处理器

2. **React 前端改动需同步检查**：
   - 类型定义（interface.tsx）
   - 默认值（defaultVar.tsx）
   - 数据流（dataContext.tsx）
   - 保存逻辑（axios/save.tsx）

### 3.3 向后兼容原则

1. **数据结构不变**：即使后端拆分，传给前端的数据格式保持一致
2. **导入导出兼容**：导出的 JSON 文件格式保持与旧版一致
3. **迁移幂等**：迁移脚本可重复执行，不会产生副作用
4. **降级回退**：关键路径必须有降级方案（如 `load_legacy()`）

### 3.4 代码精简原则

1. **数据与逻辑分离**：注册表是数据，加载器是逻辑，不要混在一起
2. **消除重复**：相同的判断逻辑提取到公共方法
3. **删除死代码**：不再使用的旧代码立即删除，不要注释保留
4. **注释精简**：只保留必要的分类注释，删除冗余的行内注释

## 四、后续优化建议

### 4.1 短期（第二阶段前）

1. **自动生成搜索索引**：从 `registry.php` 自动生成 `searchIndex`，避免双写

3. **go 中间页响应式统一**：提取公共 CSS，所有中间页共享
