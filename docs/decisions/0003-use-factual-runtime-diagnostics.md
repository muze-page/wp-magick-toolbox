# ADR-0003: 用事实型运行诊断替代综合健康分

## 状态

已接受

## 日期

2026-07-17

## 背景

原诊断摘要以 60 分为基础分，通过是否开启 SEO、对象缓存、国内 CDN、图片 Alt、搜索日志等可选功能加减分。顶层 `risks` 却只统计高风险和实验性模块，因此概览页可能同时显示“54 / 100”和“0 个风险提示”。

这些权重没有统一风险依据，也不能区分“小站不需要对象缓存”与“运行环境真的不满足要求”。摘要还返回前端没有消费的推荐、修复建议、服务提示和完整环境对象，并在处理诊断 REST 请求时同步请求一次本站 REST，增加了重复计算和最长 5 秒的阻塞风险。

当前插件没有用户和兼容负担，可以直接收缩契约，不保留旧分数字段或兼容层。

## 决策

- 诊断摘要只返回 `status`、`items`、`module_risks` 和 `generated_at`。
- `items` 只包含影响插件运行的 PHP 与 WordPress 版本检查。
- 高风险和实验性模块只在 `module_risks` 中表达一次，不再生成重复检查项。
- SEO、媒体、国内 CDN、对象缓存等可选配置不影响总体健康状态；概览页仍可将其作为可选下一步。
- 搜索统计只由搜索健康接口负责，诊断摘要不再重复读取或评价搜索数据。
- 删除综合分数、推荐、自动修复建议、服务提示、完整环境输出、未消费的 item action，以及 REST 自请求探针。
- 概览页显示真实通过数、待关注数、需处理数、模块风险数和具体待核对标题。

## 备选方案

### 重新调整分数权重

拒绝。即使重新加权，百分比仍会把不同站点策略压缩成没有统一基准的单一数字，并需要持续维护隐藏规则。

### 将分数改名为“配置完整度”

拒绝。工具箱功能是按需启用，不存在所有站点都应达到的统一完整配置；该指标仍会鼓励不必要的开关。

### 为可选项新增中性状态

拒绝。可选项已经由对应设置页和概览下一步承载；继续塞入健康诊断只会增加状态分类和重复规则。

## 影响

- 诊断 REST 契约发生一次无兼容层的清理，前后端在同一提交切换。
- 正常站点不会因为未启用可选功能而被标记为“需要关注”。
- 摘要计算不再发起网络请求，也不再重复搜索健康聚合。
- 未来新增诊断必须对应明确的运行要求或故障事实；可选配置建议不能通过分数或 warning 混入。

## 回滚

本决策以单一提交落地，可以完整 revert。回滚不得只恢复前端分数或后端旧字段之一，以免再次造成契约不一致。
