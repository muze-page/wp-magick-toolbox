## 一、迁移概述

本次迁移将插件从「先加载全部文件，再判断开关」的架构，改造为「先判断开关，只加载开启的文件」的按需加载架构。

**核心变更**：路由表机制 + 降级回退 + 前端代码分割 + 资源规范化 + 公共逻辑统一

**总修改文件数**：40+ 个 PHP + 6 个 TS/TSX + 2 个配置文件 + 2 个文档

---

## 二、架构决策与原因

### 2.1 为什么选择路由表而非动态判断？

**方案对比**：

| 方案 | 优点 | 缺点 |
|---|---|---|
| 每个模块内 `if (get_config(...))` | 简单 | 每个文件都要判断，无法缓存，性能差 |
| 路由表（最终选择） | 一次生成，多次读取，可缓存 | 需要维护映射关系 |
| 自动扫描 + 反射 | 零维护 | PHP 反射开销大，兼容性差 |

**选择路由表的理由**：
1. 生成一次（保存设置时），读取 N 次（每次页面加载）
2. 支持 WP Object Cache，缓存命中率极高
3. 降级回退机制保证零风险
4. 路由表是纯数组，序列化/反序列化成本极低

### 2.2 为什么保留 load_legacy() 降级？

- **首次激活**：路由表为空，必须回退到传统加载
- **配置损坏**：路由表数据异常时，传统加载是安全网
- **热修复**：生产环境出问题时，清空路由表即可回退，无需回滚代码
- **渐进式迁移**：可以逐步测试每个模块，不必一次性切换

### 2.3 为什么没有迁移 132 处 MaBox_Admin::get_config() 调用？

- **风险/收益比极低**：132 处分布在 30 个文件中，一次性迁移极易引入回归
- **Helpers 已就位**：新代码可以使用 `MaBox_Helpers::get_config()`
- **功能等价**：Helpers 的 get_config 本质是代理调用，行为完全一致
- **建议**：后续修改某个模块时，顺手迁移该模块的调用即可

## 三、踩过的坑

### 3.1 重复方法声明（致命错误）

**现象**：`class-magick-mixtrue-admin.php` 中出现两个 `public function load()` 方法

**原因**：新增按需加载版本的 `load()` 时，忘记删除旧版本

**后果**：PHP Fatal Error: `Cannot redeclare MaBox_Admin::load()`

**教训**：
- 替换方法时，必须先删除旧方法再添加新方法
- 使用 `grep -n "function load()"` 检查重复
- 提交前必须运行 `php -l` 语法检查

### 3.2 选项名拼写错误

**现象**：`Magick_ToolBox_Acttive_Modules` 拼成了 `Acttive`（双 t）

**后果**：路由表写入和读取使用不同键名，导致缓存永远 miss

**教训**：
- 常量名/选项名必须定义为常量，避免在代码中硬编码字符串
- 使用 `grep` 全局搜索确认一致性后再提交

### 3.3 导入/REST 保存后路由表未更新

**现象**：通过导入设置或 REST API 保存配置后，路由表仍是旧数据

**后果**：按需加载与实际配置不一致，功能表现异常

**修复**：在 `import_settings_callback()` 和 `rest_save_settings()` 中补充路由表重建逻辑

**教训**：
- 所有修改配置的入口点都必须同步更新路由表
- 使用 DRY 原则：将路由表重建逻辑提取为独立方法

### 3.4 H5 模块在降级模式缺少移动端守卫

**现象**：`load_legacy()` 中无条件加载 H5 模块

**后果**：降级模式下，桌面端也会加载 H5 相关代码

**修复**：添加 `if (wp_is_mobile())` 守卫

**教训**：
- 降级回退路径必须与正常路径行为一致
- 每个条件加载的模块，在降级路径中也要保持相同的条件

### 3.5 wp_add_inline_script 的时机问题

**现象**：在 `the_content` 过滤器中调用 `wp_enqueue_script`

**原因**：`wp_enqueue_scripts` action 在 `the_content` 之前已执行完毕

**后果**：脚本可能不会正确输出

**解决方案**：
- 对于条件性加载的脚本，直接在过滤器中调用 `wp_enqueue_script` + `wp_add_inline_script`
- WordPress 允许在 `wp_footer` 之前随时入队脚本（footer 脚本会正常输出）

### 3.6 wp_add_inline_style 的时序限制

**现象**：在 `comment_form_defaults` 过滤器中调用 `wp_add_inline_style`

**原因**：样式已在 `wp_head` 中输出，后续添加的 inline style 不会生效

**解决方案**：
- 对于必须在 `wp_head` 之后添加的样式，保留 inline `<style>` 输出
- 或者在 `wp_enqueue_scripts` 阶段预先注册样式和 inline 数据

## 四、迁移模式总结

### 4.1 PHP 资源迁移模式

#### 模式 A：简单内联 JS → wp_add_inline_script

**Before**:
```php
public static function run() {
    add_action('wp_footer', array(__CLASS__, 'js_code'));
}
public static function js_code() {
?>
    <script>/* JS code */</script>
<?php
}
```

**After**:
```php
public static function run() {
    add_action('wp_enqueue_scripts', array(__CLASS__, 'enqueue_assets'));
}
public static function enqueue_assets() {
    wp_enqueue_script(MAGICK_MIXTURE_NAME . '_handle', '', array(), MAGICK_MIXTURE_VERSION, true);
    $js = "/* JS code */";
    wp_add_inline_script(MAGICK_MIXTURE_NAME . '_handle', $js);
}
```

#### 模式 B：内联 CSS → wp_add_inline_style

**Before**:
```php
public static function run() {
    add_action('wp_footer', array(__CLASS__, 'css_code'));
}
public static function css_code() {
?>
    <style>/* CSS */</style>
<?php
}
```

**After**:
```php
public static function run() {
    add_action('wp_enqueue_scripts', array(__CLASS__, 'enqueue_assets'));
}
public static function enqueue_assets() {
    wp_enqueue_style(MAGICK_MIXTURE_NAME . '_handle', '', array(), MAGICK_MIXTURE_VERSION);
    $css = "/* CSS */";
    wp_add_inline_style(MAGICK_MIXTURE_NAME . '_handle', $css);
}
```

#### 模式 C：动态值内联 JS

```php
public static function enqueue_assets() {
    $value = esc_js(self::$option);
    wp_enqueue_script(MAGICK_MIXTURE_NAME . '_handle', '', array(), MAGICK_MIXTURE_VERSION, true);
    $js = "var config = '$value'; /* use $value */";
    wp_add_inline_script(MAGICK_MIXTURE_NAME . '_handle', $js);
}
```

#### 模式 D：条件性资源

```php
public static function enqueue_assets() {
    if (!is_user_logged_in()) {
        wp_enqueue_style(MAGICK_MIXTURE_NAME . '_handle', '', array(), MAGICK_MIXTURE_VERSION);
        $css = "/* CSS */";
        wp_add_inline_style(MAGICK_MIXTURE_NAME . '_handle', $css);
    }
}
```

### 4.2 React 组件懒加载模式

**Before**:
```tsx
import Page from "@/components/page/index";
const items = [{ key: "1", children: <Page /> }];
```

**After**:
```tsx
const Page = lazy(() => import("@/components/page/index"));
const TabFallback = <div style={{ padding: "48px" }}><Spin size="large" /></div>;
const items = [{ key: "1", children: <Suspense fallback={TabFallback}><Page /></Suspense> }];
```

```typescript

  'tab-page': ['./src/components/page/index'],
  'vendor': ['react', 'react-dom', 'antd'],
},
```

## 六、回滚方案

### 6.1 快速回滚（30 秒）

```php
// 在 WP-CLI 或 PHP 环境中执行
update_option('Magick_ToolBox_Acttive_Modules', array());
```

系统将自动回退到 `load_legacy()` 传统加载模式，所有功能恢复原状。

### 6.2 代码回滚

恢复 `MaBox_Admin::load()` 为改造前的版本，保留路由表生成逻辑以便后续重新启用。

### 6.3 数据保留

路由表数据保留在数据库中，修复问题后可重新启用，无需重新配置。

## 十、P0-P3 补充落实经验

### 10.1 预设配置 key 名对齐（P0）

**现象**：`presets.ts` 中统计功能字段设为 `true`，但实际模块需要的是统计 ID 字符串

**后果**：预设应用后，`baidu_tonji.php` 输出 `hm.js?true`，百度统计无法工作

**修复**：预设中统计字段改为占位字符串（`'请填写百度统计ID'`），既满足 `!empty()` 条件激活模块，又提示用户替换

**教训**：
- 预设配置必须与模块实际接收的数据类型一致
- 需要用户填写具体值的字段，预设中应使用占位字符串而非布尔值
- 验证预设有效性：检查 `build_active_modules_map()` 中每个 key 的判断逻辑

### 10.2 L1 资源内联化模式（P1 核心）

**问题**：大量模块使用 `wp_enqueue('handle', '', ...) + wp_add_inline_script` 模式，空 URL 导致 WordPress 注册无效脚本标签

**修复**：改为 `wp_head`/`wp_footer` 直接 echo

**Before**（7 行）：
```php
public static function run() {
    add_action('wp_enqueue_scripts', array(__CLASS__, 'enqueue_assets'));
}
public static function enqueue_assets() {
    wp_enqueue_script(MAGICK_MIXTURE_NAME . '_handle', '', array(), MAGICK_MIXTURE_VERSION, true);
    $js = "/* code */";
    wp_add_inline_script(MAGICK_MIXTURE_NAME . '_handle', $js);
}
```

**After**（5 行）：
```php
public static function run() {
    add_action('wp_footer', array(__CLASS__, 'render'), 999);
}
public static function render() {
    echo '<script>/* code */</script>' . "\n";
}
```

**适用条件**：
- 代码量 < 2KB
- 无外部文件依赖
- 无第三方库依赖

**已迁移模块**（10+ 个）：`all_grey`、`dynamic_title`、`ban_copy`、`front_debug`、`ban_open_qq`、`ban_open_weixing`、`baidu_tonji`、`unlisted_vague_img`、`copy_pop_up(concise)`、`comment_emoji(inline)`、`runcode`

### 10.3 Vite 分包策略细化（P1）

**问题**：`vite/public/vite.config.ts` 只分了 `vendor` + `share` 两个 chunk，html2canvas（~205KB）被打包进 share chunk

**修复**：新增 `poster`（html2canvas 独立）、`qrcode`（二维码独立）分包，配合 React `lazy()` 实现按需加载

**教训**：
- 重量级第三方库（html2canvas 等）应独立分包，避免污染主 chunk
- Vite manualChunks 必须配合 React lazy() 才能生效——只分包不懒加载等于没分
- public 端的 share 功能只在文章页使用，应考虑条件入队（`is_singular()` 判断）

### 10.4 copy_pop_up 语法错误修复（P1）

**现象**：文件中存在 stray `</script>` 标签（第 44 行）和两个同名 `jiub()` 方法

**原因**：历史代码复制粘贴时未清理旧版本

**后果**：PHP 语法错误，启用复制弹窗功能时页面白屏

**教训**：
- 修改前先通读文件，发现重复代码立即清理
- 同一类中不允许同名方法，PHP 会 Fatal Error

### 10.5 MaBox_Helpers 公共逻辑迁移（P2）

**迁移范围**：
| 原模式 | 新方法 | 迁移数量 |
|--------|--------|----------|
| `$_SERVER['HTTP_X_FORWARDED_FOR']` | `MaBox_Helpers::get_real_ip()` | 2 处 |
| `wp_is_mobile()` | `MaBox_Helpers::is_mobile()` | 4 处 |
| `is_user_logged_in()` | `MaBox_Helpers::is_logged_in()` | 8 处 |

**迁移后**：原模式 0 残留（仅剩 1 处注释代码）

**教训**：
- 迁移公共逻辑时，优先迁移高频使用的模式（如 `is_user_logged_in` 出现 8 次）
- `search_limit.php` 中 `$_SERVER['REMOTE_ADDR']` 也应迁移为 `get_real_ip()`，保持 IP 获取逻辑统一
- 迁移后必须全局 grep 确认无残留

## 十一、更新后的经验教训

1. **替换方法时先删后加** — 永远不要在同一类中保留两个同名方法
2. **所有配置写入点都要同步** — 路由表重建逻辑必须覆盖所有修改配置的入口
3. **降级路径与正常路径等价** — 回退模式的行为必须与正常模式一致
4. **常量优于字符串** — 选项名、配置键等应定义为常量
5. **分批迁移优于一次性** — 132 处调用证明，大规模迁移应分步进行
6. **构建产物不入库** — dist/ 目录应始终 gitignore
7. **先验证再提交** — 每个 PHP 文件修改后必须运行 `php -l`
8. **预设应用要考虑状态同步** — 深合并后必须同时更新 UI 状态和持久化数据
9. **预设值类型必须与模块接收类型一致** — 需要字符串的字段不能设为布尔值
10. **L1 资源直接 echo 优于 wp_enqueue('')** — 空 URL 的 wp_enqueue 注册无效标签，直接 echo 更简洁
11. **Vite manualChunks 必须配合 lazy()** — 只分包不懒加载，所有 chunk 仍会首屏加载
12. **重量级第三方库独立分包** — html2canvas 等应独立 chunk，避免污染主 bundle
13. **修改文件前先通读** — copy_pop_up 的 stray `</script>` 和重复方法证明，历史代码可能有隐藏问题
14. **IP 获取逻辑统一** — 所有模块应使用 `MaBox_Helpers::get_real_ip()`，避免 `$_SERVER['REMOTE_ADDR']` 和 `HTTP_X_FORWARDED_FOR` 散落各处
15. **测试先静态后动态** — 静态验证（编译检查、grep 确认）可离线完成，应先行执行
