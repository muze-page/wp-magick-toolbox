> 阶段状态：工程实现与受控开发评测已收口，等待内部真实案例验证
>
> 时间范围：2026-07-22 至 2026-07-23
>
> 适用版本：Npcink Site Toolbox 3.2.0 开发分支
>
> 关联 PR：[Add read-only DeepSeek diagnostics and follow-ups](https://github.com/npcink/npcink-site-toolbox/pull/28)

## 一、结论

本阶段完成了一个符合 Site Toolbox 定位的 DeepSeek 只读诊断闭环：

- Site Toolbox 不实现 Provider Runtime，也不依赖 `npcink-cloud-addon`；
- WordPress Connectors 管理凭据，单独安装的 DeepSeek Provider 提供模型能力；
- 插件只负责生成白名单事实、让管理员预览、显式发送、展示回答和临时追问；
- AI 不自动修改配置、不执行命令、不保存问题、快照、回答或追问历史；
- 故障排查、性能分析、维护解读、设置风险和修复复验共用同一组只读边界；
- 网页预览、Markdown 复制和发给 AI 的结构化事实是三个独立契约；
- 工程门禁、开发评测和真实案例采用分层证据，前两层不能替代真实价值验证。

当前适合停止增加 AI 功能，进入内部真实使用期。是否长期保留并扩展直连能力，
应由 10 个拥有独立根因或最终证据的真实案例决定。

## 二、最初问题与产品判断

最初需要解决的不是“能否调用 DeepSeek”，而是三个更重要的问题：

1. DeepSeek 应直接接入 Site Toolbox，还是依赖 `npcink-cloud-addon`？
2. AI 能力是否会让当前工具箱变得过重、偏离定位？
3. 故障排查和性能分析是否真的比复制报告到外部 AI 更有价值？

最终选择单插件直连，但把边界限定得很窄：

- **Site Toolbox 负责事实和交互**：采集、脱敏、预览、请求、展示。
- **WordPress AI Client 负责统一调用接口**。
- **Connectors 负责连接和凭据**。
- **AI Provider For DeepSeek 负责 DeepSeek Provider 实现**。
- **Cloud Addon 不参与**：不引入云端控制面、队列、计费、账号或跨站工作流。

这个方案值得做的前提不是“AI 功能多”，而是减少管理员在站点报告与 AI 之间手工
复制、解释字段和重复补充上下文的成本。为了避免复杂度扩散，本阶段拒绝了完整聊天、
自动修复、日志自由上传、Provider 配置页和会话持久化。

## 三、已实现的用户闭环

管理员在“关于与帮助 → AI 诊断”独立 TAB 中完成操作：

1. 选择故障排查、性能分析、维护解读、设置风险或修复复验；
2. 生成并检查语义化网页预览；
3. 需要人工降级时复制 Markdown；
4. 显式点击后，服务端重新采集或校验同范围白名单事实；
5. 通过 WordPress AI Client 固定调用 `deepseek` Provider；
6. 使用安全 Markdown 组件展示回答；
7. 在同一事实范围内临时追问，最多 3 轮；
8. 切换模式或刷新页面即清除回答、基线和追问历史。

```mermaid
flowchart LR
    A["管理员选择分析模式"] --> B["生成白名单事实包"]
    B --> C["语义化网页预览"]
    C --> D{"管理员显式发送？"}
    D -- "否" --> E["复制 Markdown 或停止"]
    D -- "是" --> F["服务端重新采集或校验"]
    F --> G["WordPress AI Client"]
    G --> H["DeepSeek Provider"]
    H --> I["安全 Markdown 展示"]
    I --> J["同一事实下最多 3 轮追问"]
```

### 三种数据形态

| 用途 | 形态 | 原因 |
|---|---|---|
| 管理员检查 | 语义化网页 | 表格、分区、字段和局限更容易阅读 |
| 人工复制与降级 | Markdown | 可移植、可审查，不依赖当前页面 |
| 实际 AI 请求 | 结构化 JSON 白名单事实 + 服务端约束 | 不把展示 HTML 或字段值当成指令 |
| AI 回答 | Markdown，经 `react-markdown` + `remark-gfm` 渲染 | 支持常用结构，但不执行原始 HTML |

## 四、架构与安全边界

### 4.1 数据最小化

- 故障、性能和维护模式只发送固定白名单分区与字段。
- 搜索健康只发送聚合计数，不发送搜索词原文。
- 设置风险由服务端依据 Schema 重新识别，排除全部凭据路径。
- 复验基线只保存在当前浏览器页面，服务端重新规范化后再比较。
- 插件不读取或保存 DeepSeek API Key。
- 问题、快照、回答、追问和基线不写入 WordPress 数据库。

### 4.2 输出安全

- AI 回答只读展示，不触发自动操作。
- 原始 HTML 不解析，回答中的链接不变成可点击跳转。
- 当前生产站只允许查看已有日志和不会写入系统的测量。
- “临时”“可逆”或“低峰期”不能成为修改生产配置的理由。
- 配置实验只能写为测试/预发布环境实验，必须明确禁止在当前生产站执行并给出回退点。
- 当前站不建议抓包、附加进程跟踪或运行 `tcpdump`、`strace`、`rndc stats` 等命令。

### 4.3 错误与不完整回答

早期真实调用暴露出两个不同问题：

- 上游网络、鉴权、限流或服务异常需要映射为稳定且不泄露上游正文的错误；
- DeepSeek 可能生成推理内容但没有最终正文，或因 token 上限中途结束。

最终处理规则：

- 空正文或 `finish_reason=length` 只自动重试一次；
- 第二次仍为空时返回专用空正文错误；
- 第二次仍因长度中断时返回 `diagnostic_ai_incomplete_response`；
- 不把推理内容当成最终回答，也不向管理员展示半截正文。

## 五、能力范围为何停在这里

本阶段从故障排查和性能分析扩展到维护解读、设置风险、修复复验，但没有继续做完整
聊天。增加的是“轻量追问闭环”，原因如下：

- 管理员常见需求是追问依据、缺少的证据和安全验证步骤；
- 追问继续使用首次分析的同一白名单事实，不开放任意上下文；
- 最多 3 轮足够完成澄清，又不需要会话表、长期记忆或配额系统；
- 新开分析即可重新采集事实，避免旧上下文逐渐偏离当前站点状态。

因此当前能力应理解为“带有限追问的只读诊断工具”，而不是站内通用 AI 助手。

| 建议改进 | 一次开发阶段扩展到五类能力，差异达到多个前后端与测试文件 | 每项能力复用了相同事实与交互合同，扩展成本看起来较低，但总体审查面仍迅速增大 | 下一阶段冻结功能；任何新增模式必须先有真实案例证明现有模式无法覆盖 |

### 做得好的地方

- 先确定产品边界，再选择技术接入方式，没有把 Cloud、Provider 和 Site Toolbox 混在一起。
- 每次扩展都复用白名单事实、权限、脱敏和非持久化合同，没有为新模式建立平行体系。
- 把网页展示、Markdown 复制和结构化请求分开，兼顾了可读性、降级能力和安全性。
- 对 502、空正文、长度中断和模型随机性都使用真实调用复现，而不是只写理论处理。
- eval-lab 意见没有直接当成结论；输入错误的一轮完整作废，有效意见逐条回到源码核对。
- Docker 失败样本没有为了得到 10/10 而简单放宽门禁；区分了真实越界和评测误报。
- 始终区分工程门禁、开发评测和真实案例，没有把受控样本写成用户证据。

### 下次重点关注

- 在批量调用随机模型前，先为评测器准备正例、否定例和测试环境例的确定性单元测试。
- 每次 eval-lab 或跨模型评审先验证输入合同、差异来源和目标提交，再阅读模型意见。
- AI 回答完整性应优先使用 Provider/AI Client 的结构化结束原因，不依赖文本尾部猜测。
- 对任何“可逆操作”先问清作用环境；生产站只读，测试环境才谈变更与回退。
- 新功能必须由真实案例缺口驱动，不因模型“还能做什么”继续横向扩展。

## 九、可复用的开发思路

### 9.1 先划责任边界，再选择依赖

“单独接入”不等于复制一套 Provider。插件可以独立提供产品能力，同时依赖 WordPress
标准 AI Client、Connectors 和独立 Provider。独立的判断标准是产品与数据控制权，
不是零依赖。

### 9.2 AI 功能的核心不是 Prompt，而是事实合同

稳定性主要来自：

- 固定白名单；
- 明确字段 ID；
- 服务端重新采集与校验；
- 把管理员问题和历史回答都视为不可信参考；
- 无证据时必须允许“无法判断”。

Prompt 只能补充边界，不能替代数据最小化和权限控制。

### 9.3 人机格式应当分层

同一份信息面向不同消费者时，不必强行使用同一种格式：

- 人需要网页结构和视觉层次；
- 复制需要 Markdown 的通用性；
- 模型需要结构化、低歧义事实；
- 模型回答可以用 Markdown，但必须经过安全渲染。

### 9.4 轻量闭环优于完整聊天

诊断场景真正高频的是“依据是什么、还缺什么、下一步怎么安全验证”。有限追问可以
解决这些问题，不必立即承担会话存储、长期记忆、配额、删除权和上下文漂移。

### 9.5 随机系统必须分层验证

推荐顺序：

1. 单元测试验证稳定合同；
2. Local 真实 Provider 冒烟验证调用链；
3. eval-lab/多模型发现边界问题；
4. Docker 重复采样验证随机输出；
5. 人工复核区分真实失败与评测器误报；
6. 真实案例验证产品价值。

跳过任何一层，都容易把“代码能跑”“模型说得像对”或“某次样本通过”误写成最终结论。

## 十、下一阶段

### 立即执行

- 冻结 AI 功能范围，不再增加模式、聊天能力或自动操作。
- 在自有站点、测试站和内部维护中等待真实事件，不为了凑数制造“真实案例”。
- 事件发生前先记录症状、已知事实和原计划，再询问 AI。
- 使用日志、测量、代码或服务状态独立确认最终根因。
- 记录首次有效线索耗时、总耗时、错误建议和是否真的减少复制/解释成本。

### 10 个案例后的决策

- 至少 6 个案例产生正确且可执行的新线索，并多数缩短首个可验证下一步时间：
  保留直连，针对真实重复缺口做小幅优化。
- 未达到门槛，但网页预览和 Markdown 复制有价值：
  保留人工分析入口，弱化或隐藏直连。
- 出现敏感信息泄露、不可逆操作诱导或稳定越过生产只读边界：
  立即停止直连试点，先修复安全门禁。
